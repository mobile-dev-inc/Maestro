Three directions for Maestro Core — which do we commit to?

All three clean up today's tech debt. They differ in how far they go toward a fast, agent-native core. My recommendation is B.

A

A — Clean up the status quo

What it is: Refactor into clean modules + typed errors + an event stream, but keep today's device model (full view-hierarchy dump + "wait to settle" polling) and keep YAML as the runner's input.

Pros

  • Lowest risk and effort
  • Incremental; preserves all current behavior
  • Fixes the "no public surface / silent breakage" tech debt

Cons

  • Does NOT fix the root cause of slow + flaky (snapshot-poll stays)
  • Not actually agent-native (still a YAML batch runner)
  • Contract is still "run(device, yaml) → events", fusing control plane with persistence format
  • Treats symptoms, not the disease
B

B — Typed control-plane core (Recommended)

What it is: The core API becomes a typed contract across 4 dimensions — inputs (command IR), observation (perceive primitives), errors (result values), trace (span stream) — carried over one persistent bidirectional channel to a THIN on-device agent. Sync is event-driven (device pushes "UI changed" / "idle") plus actionability checks; black-box by default with an OPTIONAL gray-box idle SDK. YAML / JSON / TS are persistence adapters that compile to the command IR. The batch runner is just one consumer; agents call perceive-act primitives directly via MCP.

Pros

  • Fixes the root cause — event-driven sync kills poll-and-diff → fast + reliable
  • Genuinely agent-native (fine-grained perceive-act + structured trace for self-correction)
  • Clean separation (control plane vs persistence vs runner) = flexible, simple, clear
  • YAML survives as the strongest persistence adapter (the moat)
  • Thin on-device agent is far less brittle and easy to evolve

Cons

  • Largest effort
  • Requires rebuilding the on-device agents + transport
  • A new wire protocol to design and version
  • Migration must protect the installed YAML base and the 4 existing consumers
C

C — Gray-box-first

What it is: Lean into deep in-app integration (EarlGrey / Detox / Flutter style) for maximum speed, reliability and observability — ship an SDK that apps compile in.

Pros

  • Best-in-class synchronization (exact idle, zero sleeps)
  • Richest observability (read internal app state)
  • Potentially the fastest and most reliable

Cons

  • Breaks Maestro's zero-integration "works on any app, no SDK" positioning (ruled out as the DEFAULT)
  • Per-platform and higher maintenance
  • Requires app-team cooperation
  • Alienates the install-anywhere user base
RECOMMENDATION

Go with B, and absorb C's gray-box idle mechanism as the optional opt-in path rather than the spine. That matches the constraint we set: zero-integration black-box stays the default, gray-box is available when an app opts in for extra speed/reliability.
